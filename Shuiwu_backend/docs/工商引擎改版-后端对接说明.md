## 二、接口说明

### 2.1 图片上传（无变化）

- **接口**：`POST /api/files/upload`
- **用途**：身份证正反面和补充材料均走此接口上传，返回 `file_url`
- **请求体**：`multipart/form-data`，字段名 `file`，附带 `original_filename`
- 无需后端改动

### 2.2 表单提交

- **接口**：`POST /api/business-declaration/submit`
- **认证**：`Authorization: Bearer <access_token>`
- **Content-Type**：`application/json`

## 四、字段说明与枚举值

### 4.1 核心新增字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `declaration_type` | string | 是 | 固定值 `license_application`，后端以此区分执照申请业务 |
| `license_store_name` | string | 是 | 主营业执照办理字号 |
| `license_store_name_list` | string[] | 否 | 备用字号列表，可为空数组 `[]` |
| `id_card_number` | string | 是 | 18 位身份证号（前端已做格式校验） |
| `id_card_valid_type` | string | 是 | `range`（有期限） 或 `long_term`（长期） |
| `id_card_valid_start` | string | 条件必填 | `YYYY-MM-DD`，`id_card_valid_type=range` 时必填 |
| `id_card_valid_end` | string | 条件必填 | `YYYY-MM-DD`，`id_card_valid_type=range` 时必填；`long_term` 时前端传 `"2099-12-31"` |
| `id_card_front_url` | string | 是 | OSS 上传后的身份证人像面 URL |
| `id_card_back_url` | string | 是 | OSS 上传后的身份证国徽面 URL |
| `applicant_name` | string | 是 | 申请人姓名 |
| `applicant_phone` | string | 是 | 手机号，前端已做 `^1[3-9]\d{9}$` 校验 |
| `political_status` | string | 是 | 见下方枚举 |
| `education_level` | string | 是 | 见下方枚举 |
| `email` | string | 否 | 邮箱，选填；前端已做格式校验，可为空字符串 |
| `extra_attachments` | object[] | 否 | `{ file_url: string, name: string }[]`，补充材料，可为 `[]` |
| `user_remarks` | string | 否 | 备注，可为空字符串 |
| `agree_protocol` | boolean | 是 | 前端强制为 `true`，后端可用于审计记录 |

### 4.2 `political_status` 枚举值

| 前端显示 | 后端值 |
|----------|--------|
| 群众 | `masses` |
| 共青团员 | `league_member` |
| 中共党员 | `party_member` |
| 其他 | `other` |

### 4.3 `education_level` 枚举值

| 前端显示 | 后端值 |
|----------|--------|
| 高中及以下 | `high_school` |
| 大专 | `junior_college` |
| 本科 | `bachelor` |
| 研究生及以上 | `postgraduate` |

## 五、旧字段对比（已移除）

以下字段在本次改版中**前端不再发送**，后端无需解析（历史数据仍可保留在表中）：

| 旧字段 | 备注 |
|--------|------|
| `business_name` | 已被 `license_store_name` 替代 |
| `business_license_no` | 执照办理阶段暂无此字段 |
| `business_address` | 不再由用户填写 |
| `business_type` | 不再由用户填写 |
| `business_scope` | 不再由用户填写 |
| `operator_name` | 已被 `applicant_name` 替代 |
| `operator_id_card` | 已被 `id_card_number` 替代 |
| `operator_phone` | 已被 `applicant_phone` 替代 |
| `declaration_info`（对象） | 整个对象废弃，多业务分支逻辑移除 |
| `attachments` | 已被 `extra_attachments` 替代（字段名调整，结构不变） |

## 六、后端需要做的适配

1. **新增枚举分支**：在 `declaration_type` 的处理逻辑中增加 `license_application` 分支。
2. **新字段入参校验**：按第四节的必填要求添加服务端校验；`id_card_valid_start/end` 当 `id_card_valid_type=range` 时必填；邮箱格式校验（选填，非空时校验）。
3. **数据存储**：建议在现有 `business_declaration` 表中新增列，或以 JSON 字段 `license_extra_info` 存储执照申请专属数据，避免影响旧类型数据。
4. **响应格式**：保持现有格式即可，成功返回 `HTTP 200/201`，失败返回带 `message` 字段的 JSON，前端会展示该 message。
